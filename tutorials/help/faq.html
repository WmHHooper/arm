Frequently Answered Questions

  1. Do I have to do these tutorials?

    The tutorials cover several of your weekly homework assignments, about 1/3 of your total homework grade. Check the syllabus, and do the math.

  2. 'make' is giving me a bunch of errors:
    [@myMac t1]$ make t1
gcc -c t1.s
t1.s:1:Junk character 64 (@).
t1.s:1:Rest of line ignored. 1st junk character valued 32 ( ).
...
    How do I fix it?

    The key to the problem is in your prompt. The @myMac means you are working with a copy of the tutorials on your machine, rather than the on the . Copy any edits to your account, and login to the . Similarly, errors like this:

    [@hl36 t1]$ make t1
gcc -c t1.s
t1.s: Assembler messages:
t1.s:1: Error: junk at end of line, first unrecognized character is `@'
...

    mean that you are trying ot compile on the login server, hl36, and you should login to the . An error message like this:

    3. [ ~]$ make t1
make: *** No rule to make target `t1'. Stop.

    means that your current directory, ~, is incorrect, and you should change to the t1 directory. Before you make, check your prompt: are you are logged in in the correct directory?

  3. I keep changing the program, but it runs exactly the same way. What gives?

    Be sure that the copy you are editing is the copy you are compiling. You can check from the command line by typing, for example:

    [ t1]$ cat t1.s
@ CSC 2110, Computer Organization
@ Tutorial 1: Introduction to Assembly
...
  .ident  "Aamy Aardvark"
[ t1]$

    If the file you see on the command line is different from the one you've been editing, you're working on the wrong copy.

  4. GDB doesn't recognize my breakpoints. Here's a transcript: 
    [ t1]$ gdb t1
...
t1: No such file or directory.
...
(gdb) br ftoc
No symbol table is loaded.  Use the "file" command.
Make breakpoint pending on future shared library load? (y or [n]) 
(gdb)

    Your last compilation failed, so there is no executable file to debug. Quit GDB, scroll your terminal window back to the last compilation, find out what the error was, correct the code, re-compile, and resume debugging.

    This error is less insidious that it could be; if your compilation fails, but there is an executable file left over from a previous effort, it may take you many steps of debugging before you realize you're debugging the wrong file! Before you debug, check that the last compilation succeeded.
